# AI Consensus System – Limitations

## ⚠️ Identified Limitations & Improvements Summary

The current architecture delivers high-quality consensus results but faces challenges related to **token cost, execution speed, scoring logic maturity, and system efficiency**. Below is a concise summary of identified bottlenecks and the current improvement status.

| Problem | Cause  | Recommended Fix | Work |
| ----- | ----- | ----- | ----- |
| **High Token Utilization & High Cost** | 3-stage pipeline where large prompts are reused and expanded across stages (especially full cross scoring in Stage 3).  | Reduce prompt size and avoid sending all model outputs to all models during scoring. | **M2 implemented** – Introduced **Randomized One-to-One Scoring (Stage 3)** instead of **M1 Full Cross Scoring**, significantly reducing prompt size and token cost. |
| **Lack of Parallel API Calling (Synchronous Execution)** | Initial system design executed models sequentially; parallelism was not implemented in `run_models()`. | Refactor API calls using `asyncio` or concurrent execution methods to trigger all model calls simultaneously. | Implemented Parallel AI API Calls & Async execution using **asyncio** and relative changes like adding async await in functions APIs. |
| **Slow Execution** | Pipeline contains 3 separate stages (Generation → Consensus → Scoring), each making API calls sequentially. | Introduce parallel execution and optimize stage architecture to reduce cumulative latency. | *Further optimizations can be implemented*. Although reduced pipeline timing after implementing Parallel API Calling. |
| **Simple Scoring & Weightage System** | Initial implementation phase focused on basic aggregation logic without advanced ranking strategies. | Introduce more advanced scoring mechanisms (e.g., confidence weighting, multi-round averaging, or ranking systems). | Current system uses basic weighted scoring only. *Further optimizations can be implemented*. |
| **Winner Selection in Tie Cases** | When multiple models receive the same final score, the selection logic may rely on simple ordering or default behavior. | Implement deterministic tie-breaking logic (e.g., secondary metrics, confidence score, or additional evaluation round). | Implemented using ranking (1,1,3). The system now selects a single `winner` deterministically from the first position (index 0), while all rank 1 models are included in the `winners` list for transparency and analysis. |
| **Static System Prompt / No Custom Behavior Layer** | All models currently use a fixed system instruction: `"role": "system", "content": "You are a helpful assistant."` without contextual or task-specific customization. | Introduce dynamic system prompt configuration per stage (e.g., analytical mode, critic mode, creative mode, domain-specific roles). Allow user-defined behavior profiles. | Added system_prompt_manager.py in ./hardcodes. This can use dynamic system prompt although it is in ./hardcoded as of now. We have the system to control it. *Further optimizations can be implemented*. |
| **No Context Memory or Persistent Session Layer** | Each request is processed independently without maintaining historical interaction context or learning from past outcomes. | Implement optional context memory layer (short-term + long-term), and performance feedback tracking for adaptive model weighting. | *Not implemented yet*. System is stateless per request. |
| **No Maximum Token Usage Control** | Token limits are not explicitly enforced per stage or per request, allowing prompts to grow dynamically based on model outputs. | Implement configurable `max_token_budget` per stage and total request. Add truncation, summarization fallback, or early-stop strategies. | *Not implemented yet*. System currently allows dynamic token usage without hard limits. |
| **Limited Model Diversity (3 + 1 Models Only)** | Current architecture integrates only 3 primary models (+1 optional), limiting diversity of reasoning styles and consensus robustness. | Design modular provider registry to dynamically plug/unplug models. Support N-model scaling with dynamic weighting and performance benchmarking. | Currently limited to 3 core providers (+1 optional). *No dynamic scaling implemented yet*. |
